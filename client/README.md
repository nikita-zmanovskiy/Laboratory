# FSD Template (Next.js + TypeScript)

Готовый шаблон для запуска новых проектов на базе **Feature-Sliced Design (FSD)**.

Шаблон демонстрирует не только структуру папок, но и архитектурные правила:

- как разделять ответственность по слоям;
- как оформлять публичный API каждого среза;
- как избегать хаотичных кросс-импортов;
- как развивать проект без архитектурной деградации.

## Зачем нужна такая архитектура

FSD помогает держать кодовую базу масштабируемой:

- **Предсказуемость**: любой разработчик быстро понимает, где искать логику.
- **Изоляция изменений**: правки в одном срезе меньше ломают другие.
- **Контролируемая связность**: зависимости идут по понятным правилам, а не хаотично.
- **Удобный рефакторинг**: доменные части легко переносить, заменять или удалять.
- **Онбординг**: новый участник команды быстрее встраивается в проект.

## Что уже есть в шаблоне

- FSD-слои (`app`, `pages`, `widgets`, `features`, `entities`, `shared`)
- демонстрационную страницу на главном роуте
- моковые данные и пример связки слоев

## Что внутри демо

Главная страница показывает поток:

`pages/template-home` -> `widgets/template-showcase` -> `features/project-search` -> `entities/demo-project`

Моковые данные находятся в `public/api/demo.json`, а загрузка идет через централизованный `shared/api/http`.

## Архитектура FSD в проекте

### Слои и их роль

- `app`
  Точка инициализации приложения: layout, глобальные стили, провайдеры, общая конфигурация уровня приложения.

- `pages`
  Экранные композиции. Страница собирает виджеты и задает сценарий конкретного маршрута.

- `widgets`
  Крупные самостоятельные блоки интерфейса (части страницы), которые уже объединяют фичи и сущности.

- `features`
  Пользовательские действия/сценарии: поиск, фильтрация, формы, переключатели и т.д.

- `entities`
  Бизнес-сущности предметной области: типы, модель состояния, базовый UI, доменная логика.

- `shared`
  Техническая и инфраструктурная база без бизнес-домена: API-клиент, утилиты, общий UI-kit, типы.

### Правило направленности зависимостей

Зависимости должны идти сверху вниз по слоям:

`app` -> `pages` -> `widgets` -> `features` -> `entities` -> `shared`

Слой может использовать только равный или нижележащий слой, но не наоборот.
Например, `entities` не должен импортировать что-либо из `features` или `widgets`.

## Public API срезов

Каждый срез должен иметь явную точку входа (`index.ts`), через которую он экспортирует только нужные наружу элементы.

Пример на текущем проекте:

- `entities/demo-project/index.ts` — публичные типы, store-хук, UI-компонент.
- `features/project-search/index.ts` — публичный UI-компонент фичи.

Это дает:

- **Инкапсуляцию**: внутреннюю структуру среза можно менять без каскадных правок по проекту.
- **Контракт**: другие части кода используют стабильный набор экспортов.
- **Контроль зависимостей**: проще отслеживать, что именно используется извне.

Рекомендуется импортировать срез только через его публичный API, а не через внутренние файлы.

## Кросс-импорты и ограничения

Кросс-импорты между срезами на одном слое допустимы только когда это действительно нужно и не нарушает доменную изоляцию.

Практические правила:

- не импортируйте внутренности чужого среза напрямую (`.../model/internalThing`);
- используйте только `index.ts` целевого среза;
- избегайте циклических зависимостей между соседними срезами;
- если общая логика нужна нескольким срезам, поднимайте ее в `shared` или выделяйте отдельный `entity/feature`.

Почему это важно:

- уменьшается архитектурная связность;
- легче тестировать и переиспользовать модули;
- снижается риск "эффекта домино" при изменениях.

## Состояние и данные в шаблоне

- Store состояния сущности расположен в `entities/demo-project/model` на `zustand`.
- Загрузка данных (`loadProjects`) инкапсулирована в store.
- Фильтрация (`filterDemoProjects`) также находится в store как часть доменной модели.
- HTTP-клиент централизован в `shared/api/http.ts` (axios instance).

Такой подход показывает, как держать бизнес-логику ближе к сущности, а инфраструктуру — в `shared`.

## Быстрый старт

```bash
pnpm install
pnpm dev
```

Приложение будет доступно на [http://localhost:3000](http://localhost:3000).

## Структура (FSD)

```text
src/
├── app/        # Инициализация приложения, layout, глобальные стили
├── pages/      # Страницы и композиция крупных блоков
├── widgets/    # Самодостаточные виджеты страницы
├── features/   # Пользовательские сценарии
├── entities/   # Бизнес-сущности и их UI/model
└── shared/     # Переиспользуемые утилиты, UI-kit, api, типы
```

## Полезные команды

```bash
pnpm dev
pnpm lint
pnpm build
```
